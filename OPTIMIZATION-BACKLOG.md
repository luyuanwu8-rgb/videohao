# Videohao 优化问题归档

> 协作规则：先记录问题，沟通并确认完整方案；只有在用户明确发出“统一修改”指令后，才修改业务代码、提示词或数据库。

## OPT-001：导演分镜画面数量过多

- 状态：已实施，待新任务实机验证
- 优先级：高
- 记录日期：2026-07-06

### 问题现象

当前生成结果基本接近“一句话一张图”，长文案可能产生 100 张以上的导演分镜和场景图，导致：

- 生图成本明显增加；
- 生成、审核和渲染时间过长；
- 连续内容被切得过碎，整体叙事不够连贯；
- 用户逐图检查与单图调整的工作量过大。

### 已确认根因

1. `storyboard` 按句子生成 scene，主要服务于配音和字幕。
2. `director` 根据预计口播时长和 `imageSeconds` 计算目标画面数，当前默认约 4.5 秒一张。
3. 导演提示词鼓励把长句拆成多拍，进一步放大画面数量。
4. 当前缺少可靠的总量上限和超量后的确定性收敛机制。
5. 运行时优先读取 SQLite 中的导演提示词，单独修改源码默认提示词可能不会实际生效。

### 已确认方案

采用“长度预算 + 语义灵活分镜”，不机械执行固定的一句一图，也不为了凑数量拆分长句。

1. 保留 sentence-level scene，避免影响 TTS、字幕对齐和现有时间线协议。
2. 导演画面按语义组织：
   - 人物、地点、时间、事件或情绪明显变化时换图；
   - 连续讲述同一人物、同一地点、同一件事的 2–4 句可共用一张图；
   - 重复解释、购买引导、关注话术和免责声明尽量并入相邻内容画面；
   - 开头钩子节奏稍快，正文适中，结尾稍慢。
3. 文案长度只提供画面预算：

   `参考画面数 ≈ 有效口播字数 ÷ 25`

4. 短文案允许少于 30 张；普通长文案尽量控制在 30–50 张。
5. 50 张作为安全上限，避免再次出现 100 张以上的结果。
6. 模型首次超量时携带反馈重试；重试后仍超量，由程序按相邻语义节拍合并，保证数量收敛且 sceneId 不丢失。
7. 实施时同步处理数据库中的 `director/base`、`director/health` 运行时提示词，避免源码与数据库规则不一致。

### 参考数量

| 有效口播长度 | 参考画面数 |
|---:|---:|
| 400 字 | 约 16 张 |
| 600 字 | 约 24 张 |
| 800 字 | 约 32 张 |
| 1000 字 | 约 40 张 |
| 1250 字及以上 | 最高约 50 张 |

### 实施范围

- `src/lib/steps/director.ts`
- 导演默认提示词及 SQLite 运行时提示词同步机制
- 快速制作页中的画面节奏/数量说明
- 必要的导演结果校验和测试

### 验收标准

1. 800–1250 字常规文案的导演画面通常落在 30–50 张。
2. 超长文案不会再次生成 100 张以上画面。
3. 短文案不会为了达到 30 张而机械补图。
4. 同一事件的连续句子能够合理共用画面。
5. 所有 storyboard sceneId 均被导演画面覆盖，无遗漏、无非法编号。
6. TTS、字幕、时间线和渲染链路保持兼容。
7. 页面显示的预计数量、导演实际数量和最终场景图数量口径一致。

### 本次归档不执行的动作

- 不修改导演代码；
- 不修改源码或数据库提示词；
- 不重跑已有任务；
- 不改变现有场景图或成片。

## OPT-002：成片码流严重损坏且被错误标记为完成

- 状态：已实施，待健康磁盘重渲验证
- 优先级：紧急
- 记录日期：2026-07-07
- 故障任务：`226e0173-de3d-4a34-b6f1-523bd69d1dfd`

### 问题现象

- 成片出现半屏错位、彩色竖条、块状马赛克和字幕残缺。
- 时间线与配音总长为 393 秒，最终 MP4 只有 206.76 秒。
- 页面和数据库仍把 render 标记为 `completed`。

### 已确认根因

主要根因不是源图、分镜数量或普通压缩失真，而是项目所在 E 盘在最终渲染写入期间发生硬件 I/O 错误。

Windows `System` 日志在 2026-07-07 01:37–01:54 期间记录了 22 次 `disk / Event ID 154`：

> 由于硬件错误，磁盘 1 的逻辑块地址处的 IO 操作失败。

只读磁盘映射已确认：

- 磁盘号：1
- 盘符：E:
- 型号：`ST500DM002-1BD142`
- 总线：SATA
- 容量：500GB

### 诊断证据

1. 对应 PNG 源图完整、清晰。
2. 39 个 MP3 配音文件均可无错误解码，实际时长合计正好为 393 秒。
3. 最终 MP4 的 H.264 和 AAC 均存在大量结构性解码错误，包括：
   - `Invalid NAL unit size`
   - `Error splitting the input into NAL units`
   - AAC 数据元素损坏
4. 最终 MP4 只有 4,961 帧；按时间线 393 秒 × 24fps 应约为 9,432 帧。
5. 使用当前真实渲染函数完成的 14.55 秒最小复现可正常解码。
6. 使用与故障任务相同的 73 个视频分段完成的短时最小复现也可正常解码。
7. 使用相同 Node `shell:true` 调用方式重新拼接全部 39 段配音，得到 393.043 秒且可正常解码的音频。

因此本次事故属于渲染期间底层磁盘写入故障；“73 个分镜”不是直接触发原因。

### 暴露出的软件可靠性缺口

当前渲染器把 FFmpeg 退出码 0 当作唯一成功条件：

- 成功退出时不保存或检查 FFmpeg 警告；
- 不校验最终视频时长是否接近 timeline 时长；
- 不执行最终 H.264/AAC 全量解码检查；
- 不验证最终帧数、音轨和分辨率；
- 损坏文件仍会登记 artifact，并把任务标记为完成。

### 建议的后续修复

1. 在 render 返回成功前增加强制验收：
   - `ffprobe` 校验视频、音频、分辨率、帧率和时长；
   - 成片时长与 timeline 偏差超过容差即失败；
   - 用 FFmpeg 全量解码检查，任何码流错误都判失败。
2. 采用临时文件完成渲染和验收，只有验收通过才替换正式成片。
3. 验收失败时保留诊断日志，不登记成片 artifact，不标记任务完成。
4. 渲染工作目录和正式产物迁移到确认健康的磁盘；在 E 盘硬件问题排除前停止长时间渲染。
5. 增加磁盘 I/O 错误提示，避免用户把硬件损坏误判为画面算法或压缩参数问题。

### 立即处置建议

1. 优先备份 E 盘中的项目、数据库、提示词配置和已有素材。
2. 暂停继续向 E 盘渲染大型成片。
3. 备份完成后再检查 SATA 数据线、电源线、SMART 状态和文件系统。
4. 当前损坏成片不能视为可靠产物，需要在健康磁盘上重新渲染。

### 本次排查未执行的动作

- 未修改渲染业务代码；
- 未重跑完整成片；
- 未覆盖现有损坏成片；
- 未执行 CHKDSK、磁盘修复或其他可能加重故障盘负担的操作。

## 统一修改方案 v1：先保命，再降本提效

- 状态：已按方案执行，待实机回归
- 记录日期：2026-07-07

### 建议实施顺序

1. 先修 `OPT-002` 渲染可靠性。
   - 理由：成片损坏属于高风险问题。只要没有最终验收，页面仍可能把坏文件标记为完成，后续分镜优化即使生效，也可能产出不可用视频。
2. 再修 `OPT-001` 分镜数量控制。
   - 理由：分镜数量影响成本、时长和体验，但不会像坏码流一样直接污染最终交付结果。
3. 最后补 UI 提示和验收测试。
   - 理由：让页面里的“画面节奏/数量预期”和实际导演结果一致，避免用户误以为还是固定几秒一张。

### 第一阶段：渲染可靠性修复

#### 代码目标

在渲染器返回 `ok: true` 前，增加“成片验收”：

1. 用 `ffprobe` 读取最终 MP4：
   - format duration；
   - video codec、width、height、fps、nb_frames；
   - audio codec、duration；
   - 文件大小。
2. 与 timeline 做一致性校验：
   - 视频时长应接近 `timeline.duration`；
   - 音频时长应接近 `timeline.duration`；
   - 分辨率应等于 timeline 的 width/height；
   - fps 应接近 timeline 的 fps；
   - 文件不能过小。
3. 用 `ffmpeg -v error -i final.mp4 -map 0:v:0 -f null -` 做全量视频解码。
4. 有音频时再做全量音频解码。
5. 任一校验失败，render step 返回失败：
   - 不登记 artifact；
   - 不把 render 标记为 completed；
   - 日志写清楚失败原因，如“成片时长 206.76s，预期 393.00s，偏差过大”。

#### 文件范围

- `src/hyperframes/ffmpegRender.ts`
- `src/hyperframes/render.ts`
- `src/lib/steps/render.ts`

#### 关键策略

1. FFmpeg 后端直接在 `final_out.mp4` 通过验收后，再替换 `outRel`。
2. HyperFrames 兜底后端也复用同一套验收逻辑。
3. 多动效批量渲染时，每个 `renders/<motion>.mp4` 都要先验收；第一条 copy 到 `final.mp4` 后，再对 `final.mp4` 做一次轻量验收或复用源文件验收结果。
4. 验收日志要简洁，但错误信息必须能定位是“时长不符”“解码错误”“缺音轨”“分辨率不符”还是“ffprobe 失败”。

#### 验收标准

1. 损坏 MP4 不会被标记为 completed。
2. 成片时长明显短于 timeline 时，render step 失败。
3. H.264/AAC 全量解码报错时，render step 失败。
4. 正常短视频最小复现可以通过验收。
5. mock 模式不做真实 MP4 验收，避免破坏 smoke 链路。

### 第二阶段：分镜数量控制修复

#### 代码目标

把导演分镜从“按秒数强制拆图”改成“按有效字数给预算，按语义组织画面”。

#### 建议预算函数

```text
effectiveChars = 去空白后的口播字数
targetBeats = round(effectiveChars / 25)
targetBeats = clamp(targetBeats, 8, 50)

如果 effectiveChars < 650：
  允许 targetBeats < 30
如果 effectiveChars >= 750：
  推荐区间逐步进入 30–50
```

参考区间：

| 有效字数 | 目标画面数 |
|---:|---:|
| 300 | 约 12 |
| 500 | 约 20 |
| 750 | 约 30 |
| 1000 | 约 40 |
| 1250+ | 约 50 |

#### 行为变化

1. 不再鼓励“长句必须拆多拍”。
2. 改成“同一事件连续 2–4 句可共用一图”。
3. 开头钩子可稍快，正文稳定，结尾收束。
4. 模型输出超过上限时，先带反馈重试要求压缩。
5. 重试后仍超过上限时，程序确定性合并相邻节拍，保证最终不超过上限。
6. 所有 sceneId 必须仍被覆盖，不丢句、不破坏字幕和配音。

#### 文件范围

- `src/lib/steps/director.ts`
- `src/lib/prompt-defaults.ts`
- SQLite `prompts_config` 中的 `director/base`、`director/health`
- `src/app/quick/page.tsx`
- `src/app/tasks/[id]/panels/DirectorPanel.tsx`

#### 验收标准

1. 800–1250 字文案通常生成 30–50 张图。
2. 1500 字以上文案不再生成 70–100+ 张图。
3. 短文案不强行凑 30 张。
4. 页面显示口径从“几秒/张”调整为“画面密度/数量预算”。
5. 已有任务手动编辑导演分镜的能力不受影响。

### 需要用户确认的点

1. 是否先按“渲染可靠性 → 分镜数量控制 → UI 提示”的顺序统一修改？
2. 是否接受 50 张作为普通长文案的硬上限？
3. 在 E 盘硬件问题完全排除前，是否先把新渲染输出迁移到其他健康磁盘？

## OPT-003：成片渲染提速的护栏优先方案

- 状态：第一阶段护栏已实施，尚未开启并发提速
- 优先级：高
- 记录日期：2026-07-07

### 真实环境审计结论

当前机器与运行状态：

- CPU：Intel i5-10400，6 核 12 线程。
- 内存：16GB，但审计时可用内存仅约 97MB。
- C 盘剩余约 0.66GB，不能作为渲染临时盘。
- D 盘剩余约 10.8GB，与 C 同属 240GB SSD，可谨慎作为单任务临时盘，但空间余量不宽。
- E 盘空间充足，但已出现过硬件 I/O 错误，是历史坏片主因。
- FFmpeg 4.3.1，支持 libx264/libass。
- 真实任务 `6290182a-f666-42c9-ba59-2110f89f4700` 渲染时，单个 clip 进程工作集约 220–270MB。
- 31 段图、392.6 秒视频，在第 25 段附近临时 clip 已约 670MB；完整渲染临时文件预计约 1–2GB 级别。

### 风险判断

不能直接默认开启 clip 并发，因为：

1. 当前可用内存过低，并发 FFmpeg 可能触发系统抖动或失败。
2. D 盘空间不足以无脑承接多任务/多并发临时文件。
3. E 盘有硬件 I/O 风险，继续高强度读写存在坏片或失败概率。
4. 旧代码缺少阶段耗时日志，无法判断真实瓶颈在 clip、concat、音频、烧字幕还是验收。
5. FFmpeg 后端缺少任务级渲染锁，同任务重复触发可能互删临时目录或覆盖成片。

### 已实施的第一阶段护栏

1. FFmpeg 后端增加任务级渲染锁：
   - 同一任务已有 FFmpeg 渲染时，拒绝第二个渲染实例。
2. 增加渲染临时目录配置：
   - `VIDEOHAO_RENDER_WORK_ROOT`
   - 默认仍使用任务目录下的 `renders/`，不会自动迁移到 C/D。
3. 增加临时目录安全校验：
   - 只允许清理 `ffwork-*` 工作目录；
   - 路径必须位于受控 work root 下；
   - 创建失败时释放锁并明确失败。
4. 增加磁盘空间检查：
   - 默认要求临时目录至少满足 `max(4096MB, timeline.duration * 6MB)`；
   - 可通过 `FFRENDER_MIN_FREE_MB` 调整。
5. 增加内存预警：
   - 默认可用内存低于 768MB 时写日志警告；
   - 当前阶段不因低内存直接失败，避免破坏已有单任务渲染能力。
6. 增加阶段耗时日志：
   - 自适应提亮；
   - 每个 clip；
   - clip 总阶段；
   - 图像 concat；
   - 音频拼接/混音；
   - ASS 字幕生成；
   - 烧字幕/最终编码；
   - 成片验收；
   - 成片写入；
   - 总耗时。
7. 增加验收分档：
   - `RENDER_VALIDATE=strict`：默认，完整 ffprobe + 视频/音频全量解码。
   - `RENDER_VALIDATE=standard`：ffprobe + 视频全量解码，跳过音频全量解码。
   - `RENDER_VALIDATE=fast`：只做 ffprobe/时长/轨道/分辨率，不做全量解码。

### 当前明确不做的动作

- 不默认开启 clip 并发。
- 不降低 CRF。
- 不把最终编码默认从 medium 改 fast。
- 不关闭 strict 验收。
- 不做单 FFmpeg filtergraph 大改。

### 后续提速决策门槛

只有在连续真实任务日志证明瓶颈和资源余量可控后，才考虑第二阶段：

1. 若 clip 阶段占主要耗时，且可用内存稳定 > 1.5GB，可试 `FFRENDER_CLIP_CONCURRENCY=2`。
2. 若最终编码占主要耗时，可试 `FFRENDER_X264_PRESET=fast`，但保持 `FFRENDER_FINAL_CRF=20`。
3. 若 strict 验收耗时过高，且磁盘迁移后连续成片稳定，可试 `RENDER_VALIDATE=standard`，不建议直接 fast。
4. 若 D 盘可长期保持 > 20GB 可用空间，可考虑 `VIDEOHAO_RENDER_WORK_ROOT=D:\videohao-render-tmp`；当前 10.8GB 只能谨慎单任务使用。

## OPT-004：服务进程中断后的 running 状态残留

- 状态：已实施自愈，待后续重跑验证
- 优先级：高
- 记录日期：2026-07-07
- 现场任务：`6290182a-f666-42c9-ba59-2110f89f4700`

### 问题现象

真实任务在 render 阶段中断：

- 数据库中 task/status 仍为 `running`；
- `render` step 仍为 `running`；
- 系统中已无 `ffmpeg.exe` 进程；
- 本地 Next dev server 也已退出；
- 任务目录无 `final.mp4`；
- 临时目录只生成到 `clip_0029.mp4`，缺少最后一段和最终成片。

这会导致页面长期显示运行中，但实际没有任何进程继续推进。

### 现场证据

- 任务文案约 1761 有效字；
- 新分镜规则已生效，导演输出 31 拍；
- render 启动后进入 FFmpeg clip 阶段；
- `ffwork-1783408770448` 下存在 `clip_0000.mp4` 到 `clip_0029.mp4` 和字体文件；
- 无 `clip_0030.mp4`、`video.mp4`、`voice.m4a`、`final_out.mp4`、`final.mp4`；
- 说明中断发生在 clip 阶段末尾附近。

### 根因判断

这是服务进程/开发服务器中断后，内存中的 pipeline Promise 消失，但数据库中的 running 状态没有被收敛。

此外，重新启动 dev server 时出现大量 Next/Webpack cache：

```text
invalid stored block lengths
```

说明 `.next/cache` 构建缓存也已损坏。该缓存是可再生构建缓存，已删除后重新启动服务。

### 已实施修复

1. `ensureQueueRecovered()` 启动自愈时，除了恢复 `queued` 任务外，也扫描 `steps.status='running'` 的遗留步骤。

### 追加修正：避免误伤当前活跃队列任务

首次自愈实现暴露出一个竞态：如果 `/api/tasks` 首次触发自愈时，内存队列 worker 正在处理某个任务，该任务的 step 也会处于 `running`。这不是孤儿 running，而是当前进程真实在跑。

已修正：

- 自愈扫描 running step 时，跳过 `currentTaskId`；
- 同时跳过仍在内存等待队列中的 taskId；
- 只收敛既不在当前 worker、也不在等待队列里的遗留 running step。

现场任务 `226e0173-de3d-4a34-b6f1-523bd69d1dfd` 曾被误标 failed，但 FFmpeg 实际仍在渲染；已手动恢复其 task/render 状态为 `running`，不改动任何产物，等待流水线自然完成或失败。

### 后续建议

1. 不直接复用旧 `ffwork-*` 半成品。
2. 若要继续该任务，应重跑 render，让新护栏生效。
3. 在 E 盘硬件问题排除前，不建议连续多次长时间重渲。
4. 若再次出现 `.next/cache` 损坏，优先怀疑磁盘/缓存 I/O 问题，而不是业务代码。
